# Notification — кросс-портальный агрегатор уведомлений Битрикс24

Приложение для одного «хостового» портала Битрикс24, которое собирает счётчики и контекст уведомлений с **других** порталов Битрикс24 пользователя. Решает проблему: пользователь не видит входящих сообщений / задач / лидов с второстепенных порталов, если не залогинен туда онлайн.

## Что собирает

Для каждого подключённого портала:

| Сущность | Описание |
|---|---|
| `im` | Непрочитанные чаты и сообщения |
| `notify` | Системные уведомления |
| `tasks` | Незакрытые задачи (где пользователь — ответственный) |
| `crm` | Открытые сделки + новые лиды (где пользователь — ответственный) |
| `livefeed` | Свежие записи в ленте компании |

По каждой сущности показываются последние 5 элементов с прямой ссылкой на удалённый портал.

## Архитектура

- **Backend**: Node.js 18+, Express, SQLite (better-sqlite3)
- **Frontend**: vanilla JS + Bitrix24 JS SDK
- **Хранение токенов**: AES-256-GCM шифрование на диске
- **Получение данных** (гибрид):
  - **Webhook (push)** — каждый подключённый портал получает уникальный inbound URL `/api/webhook/:portalId/:secret`. Любое событие триггерит немедленный refresh.
  - **Polling (fallback)** — фоновый воркер опрашивает порталы с настраиваемым интервалом (15–3600 сек). Если webhook не пришёл — счётчики обновятся всё равно.
- **Авторизация подключаемых порталов**: на выбор пользователя
  - Входящий webhook URL (быстро)
  - OAuth 2.0 (требует «Локальное приложение» на удалённом портале)

## Установка

### 1. Подготовка окружения

```bash
git clone https://github.com/yuri-martynenko/notification.git
cd notification
npm install
cp .env.example .env
# Сгенерируй ключ шифрования:
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
# Вставь в .env как ENCRYPTION_KEY
```

### 2. Запуск

```bash
npm start
```

Приложение слушает на `PORT` (по умолчанию 3000).

### 3. Регистрация в Битрикс24

В платформе VibeCode:

```bash
# Через MCP/CLI
vibecode app create \
  --title "Кросс-портальные уведомления" \
  --type IFRAME \
  --url "https://your-app.example.com/install" \
  --placements REST_APP_URI
vibecode app deploy --target PORTAL
```

После установки в левом меню портала появится пункт «Уведомления».

## API

| Метод | Путь | Описание |
|---|---|---|
| `GET/POST` | `/install` | Точка входа из Битрикс24 (placement entry) |
| `GET` | `/api/portals` | Список подключённых порталов |
| `POST` | `/api/portals/webhook` | Подключить портал по webhook URL |
| `GET` | `/api/oauth/start` | Начать OAuth flow |
| `GET` | `/api/oauth/callback` | Обратный вызов OAuth |
| `DELETE` | `/api/portals/:id` | Удалить портал |
| `PATCH` | `/api/portals/:id` | Включить/выключить портал |
| `POST` | `/api/portals/:id/refresh` | Принудительно обновить один портал |
| `GET` | `/api/counters` | Все счётчики пользователя |
| `POST` | `/api/counters/refresh` | Обновить все счётчики пользователя |
| `GET/POST` | `/api/settings` | Настройки (период polling, фильтр сущностей) |
| `POST` | `/api/webhook/:portalId/:secret` | Приём событий с подключённого портала |

Все API-роуты (кроме `/install`, `/oauth/callback`, `/webhook/...`, `/healthz`) требуют заголовок `X-App-Token` с подписанным сессионным токеном.

## Безопасность

- Токены OAuth и URL вебхуков шифруются AES-256-GCM на диске
- Сессионные токены подписываются HMAC-SHA256 и имеют TTL 24 часа
- Каждый inbound webhook имеет уникальный 24-байтный секрет
- HTTPS обязателен для production (за reverse proxy)
